# 执行委派规则

> 最后更新:2026-07-04

## 摘要

项目根 `../CLAUDE.md` 定义了主 session 何时应该把任务委派给 `claude-m3`、标准调用格式、以及两条必须遵守的硬约束:必须带 `--allowedTools`(否则非交互模式下会因等待人工确认而卡死),以及 M3 每次调用都是全新会话(无记忆,prompt 必须自包含上下文)。

## 一、委派触发条件

- **委派给 claude-m3**:任务进入"具体执行阶段"——按已有计划批量改文件、跑测试、修复报错、反复迭代验证
- **主模型自己做,不委派**:任务属于"制定方案 / 架构设计 / 复杂推理判断"阶段

## 二、标准调用格式

```bash
claude-m3 -p "<具体任务描述,把上下文写清楚,因为它是全新会话>" \
  --output-format json \
  --allowedTools "Read,Write,Edit,Bash"
```

拿到返回的 JSON 后,解析 `.result` 字段作为这一步的执行结果,判断是否需要进入下一轮迭代。

## 三、--allowedTools 的必要性

这条约束来自一次真实教训:用户提到"昨天 fable 说到一个问题"——工具调用在非交互模式(`-p`)下,如果需要人工确认权限,进程会直接卡住/中断,因为非交互模式无法响应确认框。落地时专门用 `claude --help` 核实了正确参数名是 `--allowedTools`(等价 `--allowed-tools`),并在 [[测试与端到端验证记录]] 里的测试 2 和端到端验证中都显式带上了这个参数,`permission_denials` 均为空,验证了这条规则的有效性。

## 四、无记忆约束与 prompt 写法

`claude-m3` 每次调用都是全新会话,不会记得之前发生了什么。委派的 prompt 里必须把当前上下文(要改哪个文件、目标是什么、上一步做了什么)写清楚,不能假设它"知道"之前的进度。端到端验证中实际使用的委派 prompt(参见 [[测试与端到端验证记录]])就是一个自包含上下文的范例——明确给出目录路径、现象描述、三步具体要求。
