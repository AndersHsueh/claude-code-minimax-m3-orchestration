# Wiki 工作流:Ingest / Query / Lint

> 最后更新:2026-07-04

## 摘要

本主题把 Karpathy LLM Wiki 模式中的三类核心操作(**Ingest / Query / Lint**)固化为可执行 checklist。每个操作都包含触发条件、具体步骤、必登 log 条目、失败处理。LLM 在对应触发发生时必须按本 checklist 推进,不可跳过。

## 一、Ingest 工作流(新源摄取)

### 触发条件

- 用户在 `raw/` 下放了**任何**新文件(`.md` / `.xlsx` / `.docx` / `.pdf` / 图片等)
- 用户在对话中说"消化一下这份资料"/"把 X 录入 wiki"/"raw 入仓"
- 用户在对话中转发了会议纪要 / 客户邮件 / 监管文件,本质等同 ingest

### Checklist(按顺序执行)

- [ ] **Step 1:识别源类型**
  - 商务对话 / 需求清单 / 客户业务样例 / 会议纪要 / 部署日志 / 培训素材 / 其它
  - 如类型不明,向用户确认一句:"这是会议纪要还是客户文档?"

- [ ] **Step 2:登记到 log.md**
  - 追加一条 `## [YYYY-MM-DD] ingest | <源文件简述>`
  - 必含字段:源文件路径、源类型、3-5 句内容摘要、关键决策点(如有)

- [ ] **Step 3:决定影响范围**
  - 问自己:这个源会影响哪些 wiki 主题?
  - 决策树:
    - 新业务模块 / 概念 → **新建** wiki 主题页
    - 已有模块的字段/流程补充 → **更新**对应 wiki 主题页
    - 仅是引用/佐证 → 在 `INDEX.md` 来源清单登记即可

- [ ] **Step 4:执行 wiki 变更**
  - 新建主题页:每页必备 `## 摘要` + `> 最后更新:YYYY-MM-DD` + 互链到至少 2 个相关主题
  - 更新主题页:保留旧内容,仅追加新段落;同步更新该页日期戳
  - 更新 `INDEX.md`:新主题入主题清单、来源清单加新源、关键事实速查表更新(如有)

- [ ] **Step 5:登记 wiki 变更到 log.md**
  - 追加一条 `## [YYYY-MM-DD] wiki | <变动摘要>`,列出被改/新增的 wiki 文件

- [ ] **Step 6:完成确认**
  - 自检三连:
    1. `raw/` 里的源文件**未被修改**(Karpathy 三层架构强约束)
    2. `wiki/log.md` 末尾多了至少 1 条 ingest + 1 条 wiki 条目
    3. `INDEX.md` 与被改主题页日期戳已同步

### 失败处理

- 源类型不明 → 停下问用户,不猜测
- 找不到合适的 wiki 主题归属 → 新建一个,不要硬塞进不相关主题
- 源文件本身错误(错别字 / 矛盾) → **不修改 raw**,在对应 wiki 主题页加注 `> 已过时 / 待澄清:YYYY-MM-DD 起 raw/X 处有 Y 问题`

## 二、Query 工作流(查询 + 回填)

### 触发条件

- 用户提问任何关于项目的事实("X 模块包含什么"/"决策 6 是什么"/"关键节点")
- 用户请求做对比、推导、汇总("X 和 Y 区别"/"汇总所有模块状态")

### Checklist

- [ ] **Step 1:决定查询模式**
  - 简单事实问答 → 直接读 INDEX → 读对应主题 → 回答
  - 跨主题综合 → 读 INDEX → 读 2-4 个相关主题 → 综合回答
  - 探索性研究 → 读 INDEX → grep log.md → 读多主题 → 回答

- [ ] **Step 2:带来源回答**
  - 回答中明确标注事实来自哪个 wiki 主题 + raw 源
  - 涉及商务决策 / 时间节点 / 模块边界的事实,必须有 `raw/` 文件做底
  - 不要凭训练数据猜测项目特定事实

- [ ] **Step 3:评估"答案是否值得回填"**
  - 决策矩阵:

    | 答案类型 | 是否回填 |
    |----------|----------|
    | 用户问的是一次性事实("X 是多少") | ❌ 不回填 |
    | 用户问的是对比/分析,结论有沉淀价值 | ✅ 回填为新 wiki 主题 |
    | 用户问的是"为什么 X 决策如此",能补全 wiki 没显式写出的推理 | ✅ 在对应主题页追加"推理与决策理由"段 |
    | 用户问的是"现状评估"(如审计),形成结构化报告 | ✅ 写为 `wiki/审计-YYYY-MM-DD-<主题>.md` |

  - **回填阈值**:任何"下次还会用到"的答案都值得回填

- [ ] **Step 4:执行回填**
  - 若是新主题:每页必备 `## 摘要` + `> 最后更新:YYYY-MM-DD` + 互链
  - 若是补充:在原主题页追加段落 + 更新日期戳
  - **必须** 追加一条 `## [YYYY-MM-DD] wiki | <回填内容简述>` 到 log.md

- [ ] **Step 5:完成确认**
  - 自检三连:
    1. 回答中每个事实点都能在 wiki / raw 中找到出处
    2. 回填的部分是否已更新 INDEX(若是新主题)
    3. log.md 是否登记了 wiki 条目

### 反模式(禁止)

- ❌ 用训练数据硬补项目特定事实
- ❌ 答案非常有用但不回填 → 散在对话历史里,下次还要重新推导
- ❌ 回填时漏写摘要或日期戳

## 三、Lint 工作流(健康检查)

### 触发条件

- **例行**:每周一次(建议周五收工前跑一次)
- **节点触发**:每次重大 ingest 之后 / 每次 INDEX.md 更新之后 / 准备正式交付物前
- **被动触发**:发现某次对话中 `[[X]]` 实际不存在 / 某页内容明显过时

### Checklist(4 类检查)

#### 3.1 断链检查

- [ ] 运行命令(注意:跳过代码块与跨域引用):

  ```bash
  cd wiki

  # 提取所有 [[]] 链接,跳过被反引号包起来的代码示例
  for f in *.md; do
    # 用 awk 跳过 ``` 代码块(允许缩进)与行内 `code` 内容
    awk '
      /^[[:space:]]*```/ { in_code = !in_code; next }
      in_code { next }
      /\[\[/ && /\]\]/ {
        # 仅处理不在行内 code 里的链接(简单启发式:跳过含 `[[]]` 的整行)
        if ($0 ~ /`.*\[\[.*\]\]/) next
        print
      }
    ' "$f"
  done | \
    grep -hoE '\[\[[^]]+\]\]' | \
    sed 's/^\[\[//;s/\]\]$//' | \
    sed 's/|.*//' | \
    awk -F'#' '{print $1}' | \
    awk -F'/' '{print $NF}' | \
    sed 's/\.md$//' | \
    sort -u > /tmp/links.txt

  ls *.md | sed 's/\.md$//' | sort -u > /tmp/files.txt

  # 加入已知跨域引用白名单(CLAUDE.md 是 schema,在仓库根,不算断链)
  cat /tmp/files.txt <(echo "CLAUDE") <(echo "log") | sort -u > /tmp/files_with_allowlist.txt

  comm -23 /tmp/links.txt /tmp/files_with_allowlist.txt > /tmp/broken_raw.txt

  # 过滤掉 raw/backend/prototypes 等明确跨域引用
  # (按你项目的跨域目录调整这个正则)
  grep -vE "^(raw|backend|prototypes|outputs)" /tmp/broken_raw.txt > /tmp/broken.txt

  cat /tmp/broken.txt
  echo "---"
  echo "真断链数: $(wc -l < /tmp/broken.txt)"
  ```

- [ ] 对每条断链决策:
  - 链接目标是真实概念 → **新建 wiki 主题页**(带摘要 + 互链 + 日期戳)
  - 链接目标是错别字 → 修正链接(不用改文件,只改引用)
  - 链接目标是 raw/backend 文件 → **不算断链**,保留跨域引用
  - 链接目标在反引号里(如 `[[X]]` 占位符示例)→ **忽略**,是文档自身的代码示例

- [ ] 修复后追加 `## [YYYY-MM-DD] lint | 修复 N 条断链` 到 log.md

#### 3.2 Orphan 检查

- [ ] 运行命令:

  ```bash
  cd wiki && for f in *.md; do
    base=$(basename "$f" .md)
    if [ "$base" = "INDEX" ] || [ "$base" = "log" ]; then continue; fi
    if ! grep -lE "\[\[$base(\||#|\]|$)" *.md >/dev/null 2>&1; then
      echo "ORPHAN: $f"
    fi
  done
  ```

- [ ] 对每个 orphan 决策:
  - 是真应该被链接的主题 → 在 INDEX.md 的"主题清单"中显式列入(INDEX 自身是兜底 inbound)
  - 是废弃主题 → 删除(删除前 grep 一下确认无引用)
  - 是临时草稿 → 标注 `> 草稿:待整合` 或删除

#### 3.3 矛盾检查

- [ ] 重点扫这些点(容易过时的):
  - 时间节点(上线日期、试运行窗口、关键里程碑)
  - 关键决策的序号与内容
  - 模块清单(业务范围 + 集成范围 + 缺漏项)
  - 工程状态(完成度、测试数、模块数)
  - 报价/合同版本号

- [ ] 发现矛盾 → 在所有冲突页面加注 `> 矛盾:YYYY-MM-DD 此页与 [[X#Y]] 不一致,待澄清`

#### 3.4 摘要 / 日期戳 / 链接密度检查

- [ ] 每页必须有 `## 摘要`
- [ ] 每页必须有 `> 最后更新:YYYY-MM-DD` 在摘要上方
- [ ] 链接密度:除 INDEX/log 外,每页至少 5 个 `[[]]` 链接

- [ ] 运行命令:

  ```bash
  cd wiki && for f in *.md; do
    sum=$(head -20 "$f" | grep -c "^## 摘要")
    dat=$(head -10 "$f" | grep -cE "最后更新[：:] *2026")
    lnk=$(grep -oE '\[\[[^]]+\]\]' "$f" | wc -l | tr -d ' ')
    echo "$f | 摘要:$sum | 戳:$dat | 链接:$lnk"
  done
  ```

### 失败处理

- 矛盾无法在 1 个会话内解决 → 列入"待澄清"主题,留待客户/团队讨论
- orphan 数量 > 5 → 暂停新建 wiki,先清理现有结构
- 断链集中在某个主题 → 该主题可能在结构上需要拆分

## 四、维护节奏

| 操作 | 触发频率 | 建议时机 |
|------|----------|----------|
| Ingest | 每次 raw 入仓 | 即时(不可拖延到下次会话) |
| Query | 每次用户提问 | 即时(回填判断随答案一起做) |
| Lint | 每周 + 重大节点 | 周五 + INDEX 更新后 + 交付物生成前 |

## 五、与其他 wiki 的关系

- [[INDEX]] — Lint 之后要回到这里更新主题清单 / 关键事实速查
- [[log.md]] — 三类操作的每一次都必登此处
- [[../CLAUDE.md]] 第 N 行的"维基规则"是本工作流的强约束来源(结构纪律 / 时间戳纪律 / 维护纪律)
- 待澄清主题(命名模式 `序号X-Y缺失主题.md` 或 `<主题>-待澄清.md`) — Lint 矛盾检查无法解决的待澄清项,归入此类"待澄清"主题

## 六、自检命令速查

```bash
# 1. wiki 摘要 + 戳 + 链接 一览
cd wiki && for f in *.md; do
  sum=$(head -20 "$f" | grep -c "^## 摘要")
  dat=$(head -10 "$f" | grep -cE "最后更新[：:] *2026")
  lnk=$(grep -oE '\[\[[^]]+\]\]' "$f" | wc -l | tr -d ' ')
  echo "$f | 摘要:$sum | 戳:$dat | 链接:$lnk"
done

# 2. 断链扫描(详见 § 3.1)

# 3. 最近 5 条 log
grep "^## \[" wiki/log.md | tail -5

# 4. raw 文件数 vs wiki 文件数 vs outputs 文件数
echo "raw: $(ls raw/*.{md,xlsx,docx} 2>/dev/null | wc -l)"
echo "wiki: $(ls wiki/*.md | wc -l)"
echo "outputs: $(ls outputs/*.{docx,xlsx,html} 2>/dev/null | wc -l)"
