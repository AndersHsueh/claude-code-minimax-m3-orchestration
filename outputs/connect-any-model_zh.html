1. 这是什么,为什么值得做

如果你在用 Claude Code(或者其它类似的 AI 编程 / 办公助手),你可能已经发现:好用的额度是有限的, 用得越勤,越快见底。这份文档要教你一件事——把"重复跑腿"的活儿,交给另一个更便宜、几乎不限量的 AI 模型去干,而把"动脑子拿主意"的部分,留给你最贵、最强的那个模型。

打个比方: 想象你请了一位很厉害、但收费很贵的总监。总监最擅长的是拿主意、想方案、看方向——这些事只有他能做好。 但如果让总监去做"改一个错别字""重新跑一遍测试""把这十份文件都调一下格式"这种重复劳动, 就是在浪费他的时间,也是在浪费你的钱。聪明的做法是:总监只管拍板,把体力活交给一位手脚麻利、 收费便宜的助理去执行。这份文档要帮你搭的,就是这样一位"助理"。
强模型(负责动脑子)
想清楚要做什么、拆解成明确的步骤
↓ 把想清楚的步骤交出去
新接入的模型(负责跑腿)
反复执行、读写文件、跑测试、改错误

搭好之后,你会多出一个新命令。你平时该怎么用 Claude Code 还怎么用;只是遇到那些"重复劳动"的活, 可以顺手改叫这个新命令去做,原来那份宝贵的额度就能省下来,留给真正需要动脑子的时刻。

2. 开始之前,你需要准备什么

你需要两样东西:

  1. 已经装好的 Claude Code 如果还没装,先去完成官方的安装步骤,确认在终端里输入 claude 能正常打开。
  2. 一个"跟 Anthropic 协议兼容"的模型服务,并且已经从这个服务商那里拿到下面三样东西。 "Anthropic 协议兼容"的意思是:这家服务商的接口,说的是跟 Claude Code 一样的"语言", 不需要额外的转换工具就能直接对接。很多模型服务商会在自己的文档里写明"兼容 Anthropic API" 或"兼容 Claude API"字样,可以以此判断。
你需要拿到的信息它是什么(打个比方)
Base URL
(服务地址)		 相当于"总部地址"——你要连接的这个服务在网络上的哪个位置。通常是一个以 https:// 开头的网址,服务商文档里会明确写出来。
API Key
(密钥 / 令牌)		 相当于"会员卡密码"——证明"这是你的账号",不能给别人看到,更不能发到公开的地方(比如发到聊天群、贴到网上代码仓库里)。
Model Name
(模型名称)		 相当于"点菜时报的菜名"——告诉系统你要用这家服务商的哪一款模型。必须跟服务商文档里写的一模一样,包括大小写和任何后缀(有些服务商同一个模型会有"标准版"和"加长上下文版"两种名字,写错了可能用到跟你以为的不一样的版本)。

如果你还没有这三样东西,去你选定的模型服务商的官网找一找,通常在"API"、"开发者中心"或 "接入指南"这类页面能找到申请入口和文档。

3. 怎么开始

把这份 HTML 文件的路径告诉你正在用的 AI 助手(比如可以直接说:"请打开这份文档, 照着里面'给 AI Agent 的精确操作步骤'那一节,帮我配置好")。它会主动向你确认上面那三个信息, 然后自动帮你把剩下的事情都搭好。

4. 给 AI Agent 的精确操作步骤

你正在读这份文档,说明有人希望你帮他把 Claude Code 接入另一个模型。请严格按照下面的顺序执行。 代码块里凡是写成 {{像这样}} 的部分,都是占位符,执行前请替换成真实值。

🔒
安全提醒: API Key 是敏感信息。确认信息时只回显前 4 位和后 4 位(例如 sk-a1***********wxyz),不要把完整密钥打印在你会长期保留的日志、聊天记录或任何会被提交到代码仓库的文件里。
  1. Step 0 · 环境检查 确认 claude 命令已安装。如果没有,停下来告诉用户先安装 Claude Code。 
    bashcommand -v claude >/dev/null 2>&1 || echo "未检测到 claude 命令,请先安装 Claude Code"
  2. Step 1 · 向用户提出下面三个问题(逐字使用)
    • "Base URL 是什么?(服务商提供的、跟 Anthropic 协议兼容的服务地址,通常以 https:// 开头)"
    • "API Key 是什么?(服务商发给你的密钥/令牌)"
    • "Model Name 是什么?(你要使用的模型名称,必须跟服务商文档里写的完全一致,包括大小写和任何后缀)"
  3. Step 2 · 生成一个简短别名(alias) 用 model_name 生成一个用于文件命名的短别名:转小写、把所有非 a-z0-9 的字符换成短横线、合并连续短横线、掐头去尾。 
    bashraw="{{MODEL_NAME}}"
alias=$(echo "$raw" | tr '[:upper:]' '[:lower:]' | sed -E 's/[^a-z0-9]+/-/g; s/^-+|-+$//g')
    如果生成结果为空或短于 2 个字符(常见于模型名是纯中文或特殊符号的情况), 追加一个问题:"能否给这个配置起一个简短的英文别名?比如 minimax、kimi、zhipu"。
  4. Step 3 · 检查是否已存在同名配置 避免覆盖用户已有的配置。 
    bashtest -f "$HOME/.claude/settings-{{ALIAS}}.json" && echo "已存在同名配置,询问用户是否覆盖或换个别名"
  5. Step 4 · 创建配置文件 新建目录并写入 ~/.claude/settings-{{ALIAS}}.json: 
    bashmkdir -p "$HOME/.claude"
    文件内容(把 {{...}} 换成真实值): 
    json{
  "env": {
    "ANTHROPIC_BASE_URL": "{{BASE_URL}}",
    "ANTHROPIC_AUTH_TOKEN": "{{API_KEY}}",
    "ANTHROPIC_MODEL": "{{MODEL_NAME}}",
    "ANTHROPIC_DEFAULT_OPUS_MODEL": "{{MODEL_NAME}}",
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "{{MODEL_NAME}}",
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "{{MODEL_NAME}}",
    "API_TIMEOUT_MS": "1200000"
  }
}
    为什么同时写了 ANTHROPIC_MODEL 和三个 ANTHROPIC_DEFAULT_*_MODEL: 不同版本的 Claude Code 读取的字段可能不完全一样,四个都写上能保证兼容性,已实测两者同时存在不会冲突。 API_TIMEOUT_MS 设置得比默认值宽松很多,因为第三方模型服务的首次响应有时会明显偏慢(实测中遇到过接近三分钟的情况)。
  6. Step 5 · 创建可执行脚本 不要用 shell 的 alias 来做这件事——AI 编程助手执行命令时,通常启动的是 非交互、非登录的子 shell,不会加载 .zshrc / .bashrc,alias 不会生效。 必须建一个真正的可执行文件。 
    bashmkdir -p "$HOME/bin"
    文件内容,写入 ~/bin/claude-{{ALIAS}}: 
    bash#!/bin/bash
exec claude --settings "$HOME/.claude/settings-{{ALIAS}}.json" "$@"
    赋予执行权限: 
    bashchmod +x "$HOME/bin/claude-{{ALIAS}}"
  7. Step 6 · 确认 ~/bin 在 PATH 里 
    bashecho "$PATH" | tr ':' '\n' | grep -qx "$HOME/bin" && echo OK || echo MISSING
    如果输出 MISSING:根据 $SHELL 判断用户用的是 zsh 还是 bash, 把下面这行追加到对应的 ~/.zshrc 或 ~/.bashrc,并提醒用户重启终端或手动 source 一下: 
    bashexport PATH="$HOME/bin:$PATH"
  8. Step 7 · 验证测试 1:文本对话 
    bashclaude-{{ALIAS}} -p "请用一句话介绍你自己" --output-format json
    检查返回的 JSON:"is_error" 应为 false,"result" 字段应有内容。 如果 PATH 还没刷新,改用完整路径 ~/bin/claude-{{ALIAS}} 测试。
  9. Step 8 · 验证测试 2:工具调用(关键一步) 在一个临时目录里测试,并且必须带 --allowedTools——否则在非交互模式下, 工具调用会因为等待人工确认权限而卡死(没有人能在非交互模式下点"允许")。 
    bashmkdir -p /tmp/model-bridge-test && cd /tmp/model-bridge-test
claude-{{ALIAS}} -p "在当前目录创建一个 hello.txt 文件,内容写 hello" \
  --allowedTools "Write" --output-format json
    不要只看 JSON 里的自我报告——独立检查文件是否真的被创建、内容是否正确, 并确认 JSON 里 "permission_denials" 是空数组。
  10. Step 9 · 向用户汇报 如果 Step 7、8 都通过,用大白话告诉用户:配置成功了,以后在任何文件夹输入 claude-{{ALIAS}} "你想让它做的事" 就能用这个新模型干活;并提醒用户以后凡是让它自动执行任务(不是打字聊天),都要带上 --allowedTools,否则会卡住。

如果测试失败,按这个顺序排查

现象先检查
命令直接报错、连不上Base URL 是否正确、是否能被访问到(可以先用 curl 单独测一下这个地址)
报错提示未授权 / 401 / 403API Key 是否复制完整、有没有多余的空格或换行
报错提示找不到模型 / 模型不存在Model Name 是否跟服务商文档里的拼写完全一致(大小写、后缀都要对)
工具调用没反应、卡住不动是否漏了 --allowedTools 参数
响应特别慢先重试一次——第三方模型服务的首次调用有时会明显偏慢,属于已知现象,不代表配置错误

5. 常见问题

这样安全吗?我的密钥会不会泄露?

配置文件保存在你自己电脑的 ~/.claude/ 目录下,不会自动上传到任何地方。 唯一需要你自己注意的是:不要把这个配置文件、或者里面的密钥,复制粘贴发到聊天群、 贴到公开的代码仓库(比如 GitHub)里。如果不小心泄露了,去服务商的后台把这个密钥 "重置"或"作废"掉,重新申请一个就行。

这个新命令会不会跟我原来用的 claude 冲突?

不会。claude 还是原来那个 claude,新建的是一个完全独立的新命令 (比如 claude-minimax),两个互不影响,你可以随时切换着用。

为什么有时候反应很慢?

不同的第三方模型服务,响应速度差别很大,尤其是第一次调用(可能要等上一两分钟)。 这不代表配置出错了,多等一下,或者换个网络环境再试。

我可以同时配置好几个不同的模型服务吗?

可以。每配置一个,就是重复一遍第 4 节的步骤,取一个不一样的别名(比如 claude-minimax、claude-kimi),它们会各自独立,互不影响。

如果我不放心 AI Agent 自动帮我配置,可以自己动手吗?

当然可以。第 4 节里的每一条,本质上都是"新建一个文件夹""写一个文件""改一下权限"这类 简单操作,即使不太懂技术,照着步骤在终端里一行一行输入命令,也能自己完成。

6. 术语表

API
软件之间互相"打电话"沟通用的一套约定好的规则,你不需要懂细节,只需要知道它是让两个程序对话的接口。
Base URL(服务地址)
你要连接的服务在网络上的"门牌号",通常是一个网址。
API Key(密钥 / 令牌)
证明"这是你本人账号"的一串字符,作用类似密码,不能给别人看。
环境变量
程序启动时可以读取的一些"设置项",这份文档里的配置文件本质上就是在预先写好一批环境变量。
权限确认框
AI 助手想要执行某个操作(比如改文件)时,弹出来问你"是否允许"的提示框。在自动化、 不盯着屏幕的场景下,必须提前用参数说明"这些操作都允许",否则程序会一直卡在这一步等你点确认。
Anthropic 协议兼容
指某个模型服务商,把自己的接口设计成跟 Anthropic(Claude 的开发商)完全一样的格式, 这样 Claude Code 就能直接跟它对话,不需要额外的"翻译层"。
